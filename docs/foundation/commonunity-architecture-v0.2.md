# CommonUnity Architecture v0.2

## Purpose

This brief captures the clarified architecture for CommonUnity after the May 2026 product strategy session. It should be treated as the source-of-truth framing for upcoming design and implementation work.

CommonUnity is not a collection of disconnected apps. It is a developmental ecosystem for helping a person translate their source pattern into lived creation, public expression, and relational unity.

The core promise is:

> A new baseline of creation, oriented toward Unity.

## Core sequence

The intended member journey is:

```text
Invitation
  → facilitated Compass process
  → Om Cipher + Living Profile
  → Studio
  → Personal Home Page
  → cOMmons / Field
```

This sequence matters. Members do not enter CommonUnity through a self-serve software funnel. They enter through an invitation-led, human-to-human facilitated Compass process. The software grows out of that relationship and source-data work.

## The five primary layers

### Compass

Compass is the initial human-facilitated onboarding and source-data process. It is the first threshold into CommonUnity and establishes the person's foundational data, context, questions, and relationship to the system.

Compass is not merely an intake form or diagnostic. It is the first act of witnessing. It prepares the member for the generation of their Om Cipher and Living Profile.

### Om Cipher

Om Cipher is the objective source-code layer. It contains fixed or foundational inputs such as:

- Birthdate and digital root
- Gene Keys and I Ching structure
- Human Design type
- Name numerology / gematria
- Seed syllable
- Captured Bhramari / Humming Bee resonance frequency
- Other measurable or archetypal starting conditions

Om Cipher generates a deterministic personal field geometry. It is not an avatar, badge, logo, or decoration. It is a compressed information fractal of the person’s source pattern: the geometry of their “given” structure.

In CommonUnity, Om Cipher should be generated at the end of the Compass process as an initiation seal and entrance marker into Studio and cOMmons.

### Living Profile

The Living Profile is the subjective/lived qualia layer. It expresses what the person has done, is doing, and will do with their foundational pattern.

If Om Cipher is the fixed source pattern, Living Profile is lived becoming. It captures projects, offerings, questions, intentions, work, creative stream, relational commitments, and the evolving meaning of a person’s life.

Om Cipher and Living Profile are two sides of the same coin:

```text
Om Cipher = source pattern
Living Profile = lived expression
```

They are generated together through Compass onboarding and facilitation, not as products of Studio. They form the fundamental information layer that informs Studio.

### Studio

Studio is the member’s higher-octave toolkit and living workspace. It is downstream of Compass, Om Cipher, and Living Profile.

Studio is where the person works with their new baseline. It should provide creative, healing, productive, and innovative tools that help the member manifest Unity in practice.

Possible Studio tools include:

- Tuner and frequency tools
- Om Cipher refinement and export tools
- AI innovation and business canvas tools
- 8 limbs of Yoga and practice design tools
- Attunement tools
- Personal homepage builder
- Creative project tools
- Healing, course, retreat, and service-development tools

The business model can include a minimal included toolkit, then advanced tools, courses, retreats, events, and services as paid upgrades.

### cOMmons / Field

cOMmons is the relational field where coherent people, projects, offerings, and intentions meet.

cOMmons should not default to a public directory or gallery of people. Its default mode should be intention-led resonance: a member offers a sankalpa, need, question, or offering, and the OM Field returns resonant presences or pathways.

A master field directory may exist for the system steward/developer and possibly for visible members later, but directory browsing is not the primary interaction model.

## OM Field and the torus threshold

The floating torus threshold in Studio should be treated as a natural OM Field interface. It already functions as an access ritual for opening the rooms of Studio.

This threshold can host:

- The member’s recorded Om/Bhramari tone
- A fresh centering practice before entering Studio
- A way to refresh or update the frequency signature
- A path into the member’s Studio
- A future path into cOMmons or other Field spaces

The OM Field should not necessarily become a separate destination. It can function as the threshold condition through which movement happens.

## Bhramari / Humming Bee activation

Bhramari pranayama, especially with Shanmukhi mudra, is a core somatic activation for Om Cipher.

The member hums into the system and produces a measurable resonance. The system captures:

- Fundamental frequency
- Dominant harmonic
- Harmonic spread
- Tonal stability
- Resonance envelope
- Possibly duration and amplitude curve

This is not merely a data field. It is a somatic offering. The person sounds themselves into the system. This makes Om Cipher more embodied, more trustworthy, and more unique than a symbolic or numerological generator alone.

The captured tone can influence:

- Sigil color through frequency-to-light mapping
- Cymatic geometry
- Motion and pulse
- Harmonic palette
- Future cOMmons attunement
- Studio entry ritual

It should not be framed as hard biometric security in v1. It is first a ritual, resonance signature, and embodied input.

## Om Cipher v1 engine model

The Om Cipher engine should become a canonical pure logic module used across CommonUnity.

```text
canonical Om Cipher engine
  → Studio preview / threshold
  → cOMmons cover / node
  → cOMmons Profile / Living Profile public expression
  → future standalone omcipher.com product
```

The engine should accept a stable input contract and return deterministic outputs, especially SVG.

### Frequency-to-color mapping

Frequency should map to visible light through octave equivalence. The captured Hz value is multiplied upward through octaves until it falls within the visible spectrum, then converted to hue.

Working examples from the recovered spec:

- 432 Hz → red-orange
- 528 Hz → yellow-green
- 639 Hz → cyan-green
- 741 Hz → blue
- 852 Hz → indigo-violet

The palette should include:

- Primary color from tonal center
- Secondary color from the perfect fifth above
- Tertiary/shadow color from the octave below

The visual language should use radial gradients, opacity gradients, and warm dark-field color discipline. It should never feel neon or arbitrary.

### Render modes

The engine should support at least two render modes:

- `full`: all layers, used in Om Cipher detail and public profile contexts
- `node`: simplified boundary/color render, used for constellation or cOMmons cover states

## Product expansion

Om Cipher can live in three product layers.

### Inside CommonUnity

The richest version, using Compass, Tuner, Bhramari, Gene Keys, Human Design, and Living Profile data. It is the member’s identity token and source geometry inside the Field.

### Standalone web app

`omcipher.com` can become a lighter self-contained experience for generating a personal sigil from name, birthdate, frequency/note, and optional Gene Keys.

Possible offerings:

- High-resolution SVG/PNG download
- Print-ready tattoo PDF
- Animated version
- Physical products such as prints, stickers, jewelry
- Paid exports or freemium model

### iOS app

A later app-store product where the sigil lives on the phone as a breathing personal symbol, lock-screen asset, meditation focus, journaling companion, AR object, and exportable tattoo artifact.

The bar for iOS should be high: the sigil must be genuinely beautiful enough for someone to consider putting on their body.

## Immediate product decisions

1. The current “sigil” should be renamed and reframed as Om Cipher.
2. The current OM badge visual is v0 and should be replaced by Om Cipher v1 field geometry.
3. Om should remain the default seed syllable, but it should not dominate the visual.
4. Om Cipher and Living Profile should be generated through Compass onboarding, not as Studio-made artifacts.
5. Studio should present them as source/orientation layers, then offer the higher-octave toolkit.
6. cOMmons should consume the public expression of these layers but should not reduce people to browseable profile cards.

## Immediate implementation priorities

1. Define the Om Cipher input contract.
2. Create one canonical Om Cipher engine module.
3. Remove or avoid duplicated inline engine copies.
4. Implement digital root, Gene Key, Human Design, gematria, and frequency mappings.
5. Support `full` and `node` render modes.
6. Wire Studio, cOMmons cover, and cOMmons Profile to the canonical engine.
7. Add tests for deterministic parity and privacy.
8. Treat Bhramari capture as a future Tuner/Om Cipher activation flow, but design the input contract for it now.
